# 数据集规划与架构升级说明

这份文档用于统一当前项目在数据生成与训练集构建上的共识，避免后续开发再次出现“生成链路”和“训练目标”不一致的问题。

---

## 1. 当前目标

我们的核心目标不是单纯“生成一些公式样本”，而是为本地部署、后续可挂载到 Ollama 的 LaTeX 公式自动纠错模型，构建一套质量稳定、分层明确、可扩展的数据体系。

当前训练目标面向简中用户，因此数据生成优先级如下：

1. 中文主版本
2. 少量中英夹杂表达
3. 未来再扩展更系统的 bilingual 数据

---

## 2. 架构升级结论

### 旧架构

之前的方向更接近：

```text
mine_sympy.py -> data_builder.py -> 本地 Ollama(Qwen 基座) 生成训练数据
```

这个架构的问题是：

- 本地 Ollama 当前挂的是基座模型，不是数据生成最优选择
- prompt、模型调用、数据管道混在一起，职责不清
- 一旦后续换生成模型，整条链路都得改

### 新架构

我们现在已经升级为：

```text
mine_sympy.py
  -> 生成标准公式底座数据
  -> 输出 standard_latex / sympy_expr / metadata

Gemini API
  -> 负责高质量生成多视角用户输入
  -> 输出 beginner / programmer / researcher 三类输入特征

data_builder.py
  -> 负责调度、限流、重试、校验、flatten、保存

本地 Ollama
  -> 不再承担主数据生成任务
  -> 用于挂载 SFT 后模型并做本地验收测试
```

### 升级后的职责划分

- `mine_sympy.py`
  - 负责“公式底座”
  - 生成标准公式与可用的 `sympy_expr`

- `data_builder.py`
  - 负责“数据管道”
  - 不再绑定具体模型提供方

- Gemini
  - 负责“高质量语言输入合成”
  - 专注生成真实用户风格输入

- Ollama
  - 负责“训练后效果验证”
  - 不是当前数据生成主力

这个拆法的最大好处是：

- 数据契约稳定
- 生成器可替换
- 测试与生产解耦

---

## 3. 数据集分层方案

正式训练集不应当是一锅炖，而应拆为三层：

### `clean`

定义：

- 高质量
- 结构正确
- 语义清晰
- 风格分层明确
- 基本不含严重噪声

作用：

- 作为模型学习“标准映射关系”的主干数据
- 保证模型先学会稳定地从模糊输入恢复正确 LaTeX

### `noisy`

定义：

- 更贴近真实用户乱输
- 引入中英混杂、缺失符号、括号不完整、变量名误写、帽子下标混乱等长尾噪声
- 但仍需保证大体上能指向正确公式

作用：

- 提升模型鲁棒性
- 缩小训练分布与真实用户输入之间的差距

### `hard`

定义：

- 面向复杂公式和难结构
- 重点覆盖积分、求和、矩阵、偏导、深度学习公式、物理长公式
- 输入表达允许更压缩、更真实、更难恢复

作用：

- 防止模型只会“简单公式纠错”
- 强化复杂公式场景下的泛化

---

## 4. 分层占比

当前建议的正式比例：

- `clean`: 50%
- `noisy`: 30%
- `hard`: 20%

也就是：

```text
clean : noisy : hard = 5 : 3 : 2
```

### 选择这个比例的原因

- `clean` 必须占主导，否则模型容易先学坏
- `noisy` 必须足够多，否则实战鲁棒性不够
- `hard` 必须保留一定占比，否则复杂公式能力会塌

这个比例适合当前项目阶段，兼顾稳定性和泛化能力。

---

## 5. 样本量规划

### 第一阶段：验证版

建议总量先做 `3000` 条左右。

分配如下：

- `clean`: 1500
- `noisy`: 900
- `hard`: 600

#### 这一阶段的用途

- 快速验证数据生成流程
- 抽查 prompt 质量
- 做小规模试训
- 低成本发现问题并返工

### 第二阶段：正式版

建议总量做 `12000` 到 `15000` 条。

以 `12000` 为例：

- `clean`: 6000
- `noisy`: 3600
- `hard`: 2400

#### 为什么不建议一开始就冲太大

- 如果 prompt 有偏差，返工成本高
- 如果类别分布不合理，浪费生成预算
- 如果真实用户风格拟合不够，训练效果会被整批拖累

---

## 6. 领域分布规划

当前训练方向应优先围绕：

- 深度学习 / 统计机器学习
- 高等数学
- 物理

剩余配额再给：

- 矩阵 / 线性代数
- 代数 / 三角

建议领域分布如下：

- `stats_ml`: 30%
- `calculus`: 25%
- `physics`: 15%
- `matrix`: 15%
- `algebra_trig`: 15%

### 说明

这个比例不是绝对数学真理，但很适合当前项目目标：

- 贴近你实际关心的科研/工程公式输入
- 保证深度学习和高数是主干
- 又不会完全牺牲基础代数与矩阵表达

---

## 7. 文件组织方案

不要把所有数据混成一个文件直接训练。

建议拆成：

- `train_clean.jsonl`
- `train_noisy.jsonl`
- `train_hard.jsonl`

最终再合并出：

- `train_full.jsonl`

### 这样拆的好处

- 每层可以独立抽查
- 哪一层质量不满意，就只重生成那一层
- 训练时可以灵活调节配比
- 后续做 ablation 更方便

## 8. 当前 20 条样本集的定位

当前已经生成并审阅过的：

- [formulas_20_gemini.json](D:\1\formulas_20_gemini.json)
- [train_gemini_20.jsonl](D:\1\train_gemini_20.jsonl)

它们的定位不是正式训练主数据，而是：

- 格式验证集
- prompt 质量验收样本
- 小规模试训种子集

### 当前样本的优点

- 结构干净
- `instruction / input / output / metadata` 明确
- 三视角分层清楚
- `output` 直接来自我们自己的标准公式 ground truth

### 当前样本的不足

- 样本量太小
- 领域配比还不够均衡
- `researcher` 风格还不够“真实草稿”
- 脏输入强度还不够

所以它适合当 `golden sample`，不适合直接作为正式主训练集。

## 9. 当前已对齐的关键原则

### 原则 1：标准公式必须由我们自己掌控

Gemini 只负责生成输入风格，不负责生成 ground truth。

也就是说：

- 输入给 Gemini 的是 `standard_latex` 等标准公式信息
- 最终训练样本的 `output` 仍然直接使用我们自己的标准公式

这样做的原因是：

- 避免模型污染标准答案
- 避免 `output` 被二次改写
- 保持数据契约稳定

### 原则 2：Ollama 不再承担主数据生成职责

本地 Ollama 当前挂载的是基座模型，它应该用于：

- 训练后的本地效果测试
- 推理验证
- 端到端验收

而不是用于现在这一步的大规模高质量数据生成。

### 原则 3：先中文主版本，后双语扩展

目前面向简中用户，所以优先做：

- 中文表达
- 少量中英夹杂

系统性的 bilingual 数据暂时后置。

---

## 10. 后续执行顺序建议

接下来建议按这个顺序推进：

1. 改造 `mine_sympy.py`
   - 增加按领域配额采样
   - 保证源公式分布更均衡

2. 生成 `clean` 小样本
   - 先做 100 到 300 条
   - 抽查风格和质量

3. 设计 `noisy` 规则
   - 加入中英混输、符号残缺、变量误写等扰动

4. 设计 `hard` 规则
   - 针对复杂公式做专项样本

5. 先完成 `3000` 条验证版
   - 小规模试训
   - 再决定是否扩到 12000+

---

## 11. 一句话总结

我们当前已经不再是“写一个脚本随便生成训练数据”，而是在构建一套：

- 有明确架构分层
- 有稳定数据契约
- 有领域配额
- 有 clean/noisy/hard 分层
- 能服务后续 SFT 与本地 Ollama 验收

的工业化数据生产流水线。
